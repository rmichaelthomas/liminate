**Status:** LOGGED — DIAGNOSTIC SNAPSHOT
**Date:** May 12, 2026
**Author:** Rob Thomas / R. Michael Thomas
**Document type:** Gap inventory — scoped to Inscript v1 dogfooding. Seven dogfooding programs (plus four probe sub-programs) were written and executed against the built interpreter (`rmichaelthomas/inscript`). Each gap is categorized as **v1-patch** (fixable without spec change), **v2-design** (requires new vocabulary or semantics), or **UX** (interpreter output quality).

## Gap Inventory

### Category: v1-patch (fixable in current interpreter without spec changes)

**Status update (May 12, 2026):** All four v1-patch items below were fixed in the same session. Test suite remains at 385 passed. Dogfooding programs re-run cleanly and each fix manifests in the expected program's trace. Fix locations are noted in each row.

| # | Gap | Surfaced By | Description | Fix |
|---|---|---|---|---|
| P1 ✅ | **`--test` flag is position-dependent and silently ignored if not first** | Program 4 | `src/inscript/cli.py:147` checked `args[0] == "--test"`. If `--test` was passed after the file path (the convention many users follow), the check failed, `auto` stayed `False`, the unknown arg was silently dropped, and the file ran without amber auto-confirm. In Program 4, this caused the mixed-precedence amber to wait on `Confirm? (y/n):`, swallow the next line of script as input, and skip the filter — producing count=4 instead of the spec-correct count=3. The `--help` text on `src/inscript/cli.py:11` writes the flag as `--test <file>`, so the documented form works; the silent-drop on `<file> --test` was the trap. | **FIXED** in `cli.py` `main()`: scan all args for `--test` (any position) and reject unknown `--flags` with exit 2. The dogfood run now produces `count = 3` on the mixed-precedence test, and `python3 -m inscript --bogus file.insc` errors with `Error: unknown flag '--bogus'` rather than silently no-op'ing. |
| P2 ✅ | **Capitalization bug in stepwise failure message** | Program 7 case 14 | Output: `Error: I completed 'filter the orders where total is above 50' but then i can't find 'missingname'. You might need to 'remember' it first. The filter has already been applied.` Lowercase `i` after `but then`. The wrapper concatenates the sub-error verbatim, and the sub-error's `I can't find...` starts with a capital `I` that loses its capitalization when not at sentence start. | **FIXED** in `interpreter.py` `_build_stepwise_failure`: the de-capitalization step now preserves a leading standalone `I ` / `I` (the pronoun) and only lowercases other leading capitals. Output now reads: *"...but then I can't find 'missingname'..."*. |
| P3 ✅ | **Canonical rendering displayed twice on amber-confirmed statements** | Program 4 | When the user confirmed an amber-precedence prompt, the interpreter re-emitted `I understand this as: <canonical>` immediately before executing the pending AST. The user already saw that canonical rendering with the amber message. Trace excerpt: `I'll read this as: filter ... or class is addendum. ... / I understand this as: filter ... or class is addendum / I understand this as: count the docs / 3` | **FIXED** in `cli.py` `display_result`: added private `_suppress_canonical` flag; the recursive call that displays the post-confirm result passes `_suppress_canonical=True`. Trace is now: `I understand this as: filter ... / I'll read this as: (...) / count the docs / 3` — no duplicate canonical line. |
| P4 ✅ | **`show each` (or similar) outside a `where` clause raises a reserved-word-in-name error** | Program 5c | Input `show each` produced: *"The word 'each' is reserved in Inscript — it's used as a verb. Please choose a different name."* But the user did not attempt to *name* anything — they tried to *use* `each` as a value/pronoun outside its valid context (per v1b §37, `each` as pronoun is only valid inside `where`). The error category was wrong: this is a parse failure from a vocabulary-tagged token in an invalid position (v1c §46 territory), not a NAME-position violation. | **FIXED** in `parser.py` at three sites — `_consume_target` (shared by count/combine/filter) plus the inline target checks in `_parse_show` and `_parse_each`. Reserved words in target position now get *"…must be names you've created with 'remember' or 'gather'"*, and `each` gets its own message: *"'each' is a verb in Inscript — it iterates a list, or acts as a self-reference pronoun inside a 'where' clause. It can't appear as a target on its own."* |

### Category: v2-design (requires new vocabulary, semantics, or spec work)

| # | Gap | Surfaced By | Description | Design Input |
|---|---|---|---|---|
| D1 | **No way to display multiple fields per record in `each`** | Program 5b | `each the docs show words and class` → parse error `"I didn't expect 'and' here."` v1 has no construct for displaying two fields per iteration step. The user must run two `each` passes, which interleaves output by *field* rather than by *record* — exactly the opposite of what a tabular reader wants. None of the four `and` meanings in §21 covers "list of fields in a `show` clause." | Either (a) add a fifth meaning for `and` inside a `show` clause (list of fields), or (b) add a `with` clause to `each` (`each the docs show with words and class`), or (c) introduce a separate verb for tabular display. The word salad test (§20) should govern. |
| D2 | **Destructive `filter` makes lists single-use; rebuilding requires re-typing every item** | Programs 2 and 4 | To run multiple independent filter probes on the same list, the only v1 workaround is to repeat the entire `remember a list called X with item-1 and item-2 and ...` statement before each filter. In Program 4 this was tolerable (4 items). In Program 2 (30 items), it would have meant re-typing all 30 names per filter. The `remember the copy called X from Y` form copies the list but is still destroyed on filter. | Either (a) make `filter` non-destructive and add a separate `narrow` or `keep` verb for in-place mutation, or (b) add a `from` snapshot — `filter the docs where ... from docs-backup` — that filters a copy. Currently the user gets the worst of both: in-place mutation *and* no easy way to revert. |
| D3 | **Destructive named compositions silently no-op on repeated calls** | Program 3 | A composition whose body is `filter the docs where words is above 5000` works on the first call. On the second call against the now-filtered list, the predicate is still satisfied by every remaining record — the call appears to succeed but does nothing. No signal to the user that the second invocation is a no-op or that the list has already been narrowed. This is a *quiet correctness hazard* for non-programmers who would naturally expect "call the function again" to behave like calling it on the original data. | Tie into D2 — non-destructive filter would resolve this entirely. Alternatively, named compositions could capture a snapshot at definition time and always apply against the original data (the "pure function" interpretation of `remember how to ...`), at the cost of changing call-time name resolution (§23 line 464). |
| D4 | **No field-access expression for a single record** | Cross-program observation | Given `remember an order called o1 with total as 75 and status as active`, the user can `show o1` (which prints `total: 75, status: active`) but cannot `show o1's total` or `show total from o1` or `show order1.total`. The only way to access one field is to wrap the record in a list and run `each ... show total` — overkill for one record. | Either add a field-access prose form, or accept this as intentional (records are addressed only via collections in v1). Worth deciding explicitly. |
| D5 | **Composition chaining (`X from Y`) is v2-deferred but error gives no v2 signal** | Program 3b | `find-large from docs` produces: *"Error: I didn't expect 'from' here."* The spec (v1b §41, v1b §43) defers this specific construct to v2 alongside composition parameters (Q9). The error message gives no hint that this is a deferred feature versus a typo. | Add a guarded error path: when the parser sees `<composition-name> from ...`, emit a clearer message like *"Composition chaining isn't supported in v1. Call '<composition-name>' on its own line and ensure the data is already named correctly."* |
| D6 | **User-supplied descriptor disappears from the canonical rendering** | Programs 1, 2, 6 | Spec v1b §36 says descriptors between article and `called` are decorative. Implementation strips them and replaces with `value`/`record`/`list`. So `remember a domain called mobius with docs as 91 and words as 381476` is echoed as `remember a record called mobius with docs as 91 and words as 381476`, and `remember the total called sum from combine the nums` is echoed as `remember a value called sum from combine the nums` (note: descriptor *and* the article `the` are normalized). For a prose-as-syntax language, replacing the user's domain term (`domain`, `class`, `order`, `total`) with internal type words (`record`, `value`) is a small but recurring loss of voice. | Either preserve the user's descriptor in the canonical rendering (the AST already discards it, but the renderer could keep it), or document the normalization explicitly in §33 so users don't expect their words back. The former is closer to the prose-as-syntax invariant. |
| D7 | **Single-token strings preclude domain-natural language for class/category names** | Cross-program | The Möbius corpus has natural categories like `gap-inventory` (hyphenated, survives v1) but also `interactive narrative`, `clinical protocol`, `in progress` (multi-word) which cannot be expressed. Per v1d §61 this is logged for documentation. Confirmed in practice: any business-rules domain pack will run into this immediately. | Multi-word strings via quoting *or* hyphen-normalization (e.g., a stylistic rule: `interactive-narrative`). The hyphen workaround was used throughout this dogfooding pass (`gap-inventory`, `nums-copy`, `find-large`) and is comfortable, but it changes the prose. Worth deciding which path. |
| D8 | **No way to construct a list as a range of items (only ranges of numbers)** | Program 2 | The 30-item `and` chain in Program 2 is functional but tedious. `gather` builds *number* ranges; there is no analogue for "a list of items 1 through N." This isn't a v1 bug — it's a design question. | Either (a) accept that all non-numeric lists are explicit, (b) introduce a corpus-import construct in v2 (deferred under "external data sources"), or (c) allow `gather` to take a list of names rather than only numbers. |

### Category: UX (interpreter output quality, error messages, display)

| # | Gap | Surfaced By | Description | Suggested Improvement |
|---|---|---|---|---|
| U1 | **"I understand this as:" prefix on every statement is excessive for canonical-order text input** | All 7 programs | For the v1 text interpreter, virtually every line typed by the user is *already* in canonical order. The reorderer typically does no work, and the canonical rendering is identical to the input. Still, every statement (including `count the domains`, which has nothing to reorder) emits a canonical line. In Program 1 the trace is 80 lines for 45 statements. In Program 2, 30 of the first 31 output lines are `I understand this as: remember a record called doc-N ...` with no useful information. The signal-to-noise ratio collapses for any non-trivial program. | (a) Suppress the canonical line when the rendering equals the original input. (b) Add a `--quiet` or `--brief` flag for trace-suppressed runs. (c) Only emit canonical when the reorderer changed token order *or* when the user is in REPL mode. |
| U2 | **Schema-mismatch error doesn't name the offending record(s)** | Program 7 case 12 | `filter the mixed-records where total is above 50` against a list where `item1` lacks `total` produces: *"Not every item in 'mixed-records' has a field called 'total'."* For a 100-record list, the user must manually find which records are missing the field. | Include the first offender (or the first few) in the message: *"'item1' in 'mixed-records' doesn't have a field called 'total'."* The semantic analyzer already iterates the list to discover the mismatch — it has the record name in scope. |
| U3 | **"Not every item" is awkward phrasing when the count is zero or near-zero** | Programs 7 cases 6 and 12 | The same message fires for *"no item has it"* and *"some items have it but one doesn't,"* but the natural reading differs. *"Not every item in 'docs' has a field called 'nonexistent'"* sounds like *most do, but not all* — when in fact none do. | Two variants: *"No item in 'X' has a field called 'Y'"* when zero have it; *"Some items in 'X' are missing a field called 'Y' (first: '<name>')"* when partial. |
| U4 | **Multi-statement programs produce dense, hard-to-scan output** | All programs | Each statement produces 1–3 lines of trace (`I understand this as: <line>` + optional output + occasional amber). With 30+ statement programs (Program 2), the eye has to hunt for the actual data results. Section dividers, blank-line preservation, or coloring would help — currently blank lines in the source file are silently dropped (per v1c §48), so visual grouping the user typed is lost. | (a) Preserve blank lines in trace output (mirror them between statement groups). (b) Color or indent the canonical lines differently from data output. (c) Combined with U1, suppress canonical for unchanged renderings — then the trace is just data. |
| U5 | **`gather` auto-show on a 5,000-item range would emit 5,000 lines** | Inferred from spec (not tested directly to avoid noise) | `gather` always auto-shows (per v1b §40). Combined with the 10,000-item range cap (v1d §63), a perfectly valid `gather the numbers from 1 to 9999` would produce a 9,999-line comma-separated display. Useful for small ranges; punishing for large ones. | Either truncate the auto-show output above a threshold (`1, 2, 3, ..., 998, 999, 1000`), or make auto-show optional for `gather` (revert to the §24 pre-resolution behavior where naming suppresses auto-show). |
| U6 | **No simple way to display a small heading or label between sections of output** | All programs, especially Programs 4 and 7 | When running a multi-step program, there is no way to label sections: *"Section A: counts before filter"*, *"Section B: counts after filter"*. v1 has no string-literal support beyond single-token bare words, and `show` of a multi-word string is impossible (D7). | Tied to D7 (multi-word strings). Without that, the user has to interleave bare-word `show` calls to mark sections, which is itself confusing. |

## What Worked Well

The following surfaces of the v1 interpreter performed correctly and felt natural to use:

- **Auto-show for `count`, `combine`, `gather`** is exactly the calculator-like behavior described in §24. No surprise, no friction.
- **`remember ... from <verb phrase>` recursive descent** worked for `combine`, `count`, and simple name references (Program 6). The §43 disambiguation rule is invisible to the user — which is the right outcome.
- **`each ... show` with no field on a record list** auto-displays the full record as `field: value, field: value` — sensible, undocumented-but-natural extension of the `each ... show` on flat lists rule (Program 5).
- **`is X` (bare) versus `is equal to X` (long form)** both produce string equality with identical results (Program 2). The §21 dual-role rule for `is` correctly resolves to equality based on the next-token category.
- **`not above`, `not below`, `not equal to`** all produce the spec'd semantics (Program 4) — note the boundary-inclusive variants (`not above 5000` means ≤ 5000) match §21's documented intent.
- **Mixed `and`/`or` amber prompt** fires correctly with the parser's parenthesization shown (Program 4): *"I'll read this as: (class is checkpoint and words is above 5000) or class is addendum."* Once `--test` is positioned correctly (P1), auto-confirm produces the expected `(A AND B) OR C` result.
- **Three-way `and` chains in a single `where` clause** do *not* trigger amber (Program 4 final test), confirming the §30 rule that single-operator chains are unambiguous.
- **Stepwise failure** preserved the destructive filter's side effect and surfaced both the success and the failure in the same message (Program 7 case 14). This matches v1d §56 precisely.
- **All 14 error categories from v1c §50 and v1d §65** triggered correctly, with messages that lead with the problem and (in the descending-range case) include the corrected form (`Try: gather the numbers from 1 to 10`).
- **Composition definition with grammar validation at definition, name resolution at call** (Program 7 case 13): `remember how to show-missing: show ghost` defined without error; `show-missing` failed at call time with the missing-name error. Matches §23.
- **30-record list construction with a 30-element `and` chain** (Program 2) parsed and executed without performance complaints. Long `and` chains are not a v1 bug, just a usability question (D2/D8).

## Resume Prompt

*We are resuming from the Inscript v1 Dogfooding Gap Inventory (May 12, 2026). The v1 interpreter is built and passes 48 test sentences. Seven dogfooding programs (plus four probe sub-programs) were written and run against real data (the Möbius Paradigm corpus — 107 documents, 462,640 words). The gap inventory categorizes findings as v1-patch, v2-design, or UX. **18 gaps total: 4 v1-patch (all fixed in-session), 8 v2-design, 6 UX.** All four v1-patch items — P1 `--test` position-independence, P2 stepwise-message capitalization, P3 duplicate canonical on amber-confirm, P4 reserved-word-in-target error — were fixed the same day; the 385-test suite still passes and the dogfooding programs verify each fix end-to-end. The three most structural remaining gaps are: **(1) D2/D3 — destructive `filter` (and destructive compositions) make lists single-use; rebuilding requires re-typing**, **(2) D1 — `each ... show` cannot display multiple fields per record, blocking tabular display in the most common business-rules use case**, and **(3) U1 — the "I understand this as:" prefix on every statement is excessive for canonical-order text input and collapses the signal-to-noise ratio of long traces**. Cross-checks (two paths to the 462,640 corpus total; manual vs. interpreter counts on filtered subsets) all held. Next step: use the v2-design items as inputs to the v2 specification work, especially the destructive-filter question (D2) which downstream-affects compositions (D3) and the language's overall reusability story.*
